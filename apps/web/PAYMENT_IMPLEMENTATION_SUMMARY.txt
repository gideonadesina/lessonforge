========================================
HOW IT WORKS: TEST SCENARIO
========================================

TEACHER BUYS PRO PLAN:
- Starting balance: 8 credits
- Pro plan: ₦5,000 = 30 credits
- Paystack reference: "pay_abc123def456"

FLOW:
1. Frontend calls /api/paystack/initialize with plan="pro"
2. Backend initializes Paystack transaction
3. User pays ₦5,000
4. Paystack sends reference to frontend

PATH A: VERIFY ENDPOINT
GET /api/paystack/verify?reference=pay_abc123def456
 > Calls Paystack API to verify
 > Calls processTeacherPayment(reference, userId, plan="pro")
   > Records payment transaction (first time)
   > Grants 30 credits (8 + 30 = 38)
   > Marks transaction processed=true
 > Updates profile.plan = "pro"
 > Returns success with creditsAwarded=30, previousBalance=8, newBalance=38

PATH B: WEBHOOK (Paystack server)
POST /api/paystack/webhook
 > Verifies signature
 > Extracts reference, userId, plan
 > Calls processTeacherPayment(reference, userId, plan="pro")
   > Finds payment transaction already recorded
   > Checks processed flag
   > If processed=true: returns alreadyProcessed=true (no double-credit)
   > If processed=false: grants credits, marks processed=true
 > Updates profile.plan = "pro"
 > Returns 200 with {received: true}

RESULT EITHER WAY:
- Credits: 8 + 30 = 38 ✅
- Not 8 + 30 + 30 = 68 ✅ (idempotency works)
- Can run verify many times: still 38 ✅
- Can run verify + webhook: still 38 ✅

========================================
KEY TECHNICAL DETAILS
========================================

IDEMPOTENCY MECHANISM:
teacher_payment_transactions.reference UNIQUE
 - First insert with same reference fails gracefully
 - Retry finds existing record
 - Check processed flag to decide if credit already granted

ATOMIC INCREMENT (prevent race conditions):
 1. SELECT current balance (example: 8)
 2. Compute new balance (8 + 30 = 38)
 3. UPDATE with condition: 
    SET credits_balance=38 
    WHERE id=? AND credits_balance=8
    -> Only succeeds if balance is still 8
    -> If failed, retry from step 1
 - Prevents concurrent requests from adding twice

MARKED AS PROCESSED:
 - After credits granted, teacher_payment_transactions.processed=true
 - This is the permanent record
 - Prevents webhook retry from re-granting

PROFILE METADATA SEPARATE:
 - Profile.plan updated independently
 - Profile.credits_balance changed only via atomic increment
 - Both can happen without interference

WHAT'S PROTECTED
========================================

✅ Double-crediting from duplicate requests
   - Reference uniqueness prevents re-processing

✅ Race conditions (concurrent payment processing)
   - Compare-and-swap prevents balance corruption

✅ Webhook retry storms
   - processed flag prevents re-granting

✅ Verify + webhook both running
   - Only one can mark processed=true first
   - Second one sees alreadyProcessed and exits

✅ Database failures during credit increment
   - Compare-and-swap fails cleanly
   - Transaction not marked processed
   - Safe to retry later
